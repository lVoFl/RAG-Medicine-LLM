\chapter{系统设计与实现}

针对传统医疗咨询方式在获取便捷性、响应及时性以及信息一致性等方面存在的不足，以及通用大语言模型在医学问答场景下存在的幻觉问题和安全性风险，本章基于前文提出的检索增强生成方法与模型优化策略，设计并实现了一套面向“三高”慢性病管理的医学问答系统。该系统通过引入外部医学知识库与模型偏好优化机制，实现对用户问题的高效响应与相对可靠的内容生成，从而为用户提供更加便捷、规范且具有参考价值的健康咨询服务。

\section{系统需求分析}

% “三高”慢性病问答场景具有专业术语密集、咨询频率高、回答风险敏感和交互连续性强等特点，因此系统设计不能只关注“能否生成回答”，还必须同时考虑回答是否可靠、证据是否可追溯、服务是否能够稳定运行，以及后续知识更新和模块扩展是否方便。从实际使用场景出发，本文将系统需求概括为功能需求与非功能需求两个层面。

\subsection{功能需求}

结合“三高”慢性病问答场景的实际使用需求，本文将系统功能需求概括为以下几个方面：
\begin{enumerate}[label =(\arabic*)]
    \item \textbf{自然语言问答功能。}系统应能够接收用户以自然语言形式提出的健康咨询问题，并对问题内容进行理解，生成语义完整、表达清晰的回答。这是系统面向用户提供服务的基础功能，也是后续检索增强与模型推理的前提。
    \item \textbf{多轮会话管理功能。}慢性病咨询具有较强的连续性，用户往往围绕同一主题进行追问，因此系统不仅要支持单轮问答，还应能够保存历史对话内容，并结合上下文理解当前问题的真实意图，以保证回答过程的连贯性和上下文一致性。
    \item \textbf{知识检索与证据支撑功能。}针对医学问答对准确性和可靠性的需求，系统应能够在回答生成前，从医学知识库中检索出与问题相关的参考资料，并将检索结果作为模型生成回答的重要依据，从而提升回答的专业性和可信度。
    \item \textbf{引用展示功能。}系统在返回答案的同时，应能够展示回答所依赖的文档来源或参考片段，使用户能够进一步了解回答依据，提高系统输出的可核验性与可解释性。
    \item \textbf{知识库维护与后台管理功能。}为了满足系统长期运行需要，还应支持知识文档入库、索引更新、会话记录保存以及管理员后台维护等功能，使系统能够适应医学知识持续更新和后续扩展的需求。
\end{enumerate}

\subsection{非功能需求}

除功能层面的要求外，医学问答系统还需要满足一定的非功能需求，以保证其在真实场景中的可用性和稳定性。本文认为主要包括以下几个方面：
\begin{enumerate}[label =(\arabic*)]
    \item \textbf{准确性与安全性。}医学问答不同于一般信息问答，回答内容若出现事实性错误、表述失当或风险边界不清，可能对用户产生误导。因此，系统应尽量降低模型幻觉带来的影响，使生成内容与外部知识依据保持一致，并在高风险问题上体现必要的审慎性。
    \item \textbf{可解释性。}系统不仅需要给出回答结果，还应能够说明回答所依赖的知识来源，使用户能够了解答案并非完全来自模型参数记忆，而是建立在外部医学资料支撑的基础之上，从而增强系统的可信度。
    \item \textbf{响应效率。}在实际交互过程中，用户通常希望在较短时间内得到反馈，因此系统在知识检索、结果重排、上下文构建和模型推理等环节都应具备较好的处理效率，并提供流式传输的功能，以满足在线问答场景下的实时性要求。
    \item \textbf{可维护性。}医学知识内容具有持续更新的特点，系统应便于进行知识文档补充、索引重建和服务维护，以保证系统能够随着知识更新而不断完善。
    \item \textbf{可扩展性。}考虑到后续Embedding模型、重排模型和生成模型均可能被替换或升级，系统架构应尽量保持模块间的相对独立，使系统能够方便地进行功能扩展和技术迭代。
\end{enumerate}

\section{开发技术选择}

在医学问答系统的开发过程中，技术选型不仅关系到系统功能能否顺利实现，也会直接影响后续模型部署、服务联调、界面交互和知识库维护的效率。目前面向智能问答系统的实现方式通常包括一体化桌面应用、传统前后端分离式Web应用以及将模型服务独立部署的分层式Web架构等几种思路。桌面应用能够在本地集成较完整的交互界面，但往往对用户运行环境和本地算力依赖较强；传统Web框架虽然具备成熟的服务组织能力，但如果将业务逻辑、模型推理和知识检索全部耦合在同一服务中，后续在GPU资源调度、模型迭代和接口维护方面容易增加实现复杂度。考虑到本文系统既涉及多轮问答交互，又依赖检索增强和大模型推理能力，因此采用分层式的Web系统实现方案更为合适。

在前端技术选择方面，常见方案包括Vue、React以及基于模板的服务端渲染框架。Vue框架上手相对较快，适合中小型交互页面开发；传统模板渲染方式实现简单，但在复杂状态管理和流式交互展示方面灵活性相对不足。相比之下，React采用组件化开发思想，在多轮会话展示、消息局部刷新和前端状态复用等方面具有更好的适应性，更适合构建具有持续交互特征的问答界面。本文系统需要实现历史会话切换、回答流式输出、引用证据展示以及后台知识管理等功能，因此最终选用React作为前端实现框架，以提高界面组织能力和交互扩展能力。

在业务后端技术选择方面，常见方案包括Flask、Spring Boot以及Node.js生态下的Express等。Spring Boot工程体系成熟，但整体配置和学习成本相对较高；Flask轻量灵活，适合封装模型推理接口，但若同时承担认证、会话管理和复杂业务编排，开发效率和异步处理优势并不突出。考虑到本文系统需要频繁处理前端请求转发、会话读写、流式响应和模型服务编排等任务，因此业务层选择基于Node.js的Express框架实现。Node.js在高并发I/O和流式通信方面具有较好适应性，能够较方便地完成SSE数据转发、接口组织和多模块协同，从而满足在线问答场景下的后端服务需求。

在模型服务与数据支撑技术选择方面，本文将模型推理部分独立部署为Python服务，并采用Flask对外提供接口。这样做的原因在于，Embedding模型、重排模型和生成模型的调用均依赖Python生态，若直接在业务后端中嵌入模型推理流程，不仅会增加服务耦合程度，也不利于GPU资源的独立管理。将模型推理单独封装为Flask服务后，一方面能够便于加载和管理深度学习模型，另一方面也有利于后续替换模型版本或调整RAG链路。

数据存储方面，系统选用PostgreSQL保存用户、会话、消息和文档元信息，并利用其对JSONB字段的支持存储半结构化问答内容；在知识检索方面，系统采用FAISS构建向量索引，以满足医学文本语义检索的效率需求。

\section{系统总体架构}

本系统整体架构采用分层式设计，将交互逻辑、业务逻辑、模型能力与数据能力进行划分，各模块之间相对独立，能够在一定程度上实现故障隔离。系统架构如图\ref{fig:framework-structure}所示，主要由四个层次构成。

前端应用层只承担交互和状态管理，不直接参与模型推理，具体包括用户登录、会话管理、消息渲染、流式展示、引用证据呈现以及管理员知识库入口等功能。

业务服务层基于Node.js，采用Express框架构建统一的 API 网关，用于对外提供服务接口。业务服务层主要负责身份认证与权限控制、会话上下文组织、消息持久化、模型服务编排和知识库维护。

模型推理层是一套独立的Python服务，部署在高性能GPU服务器上，基于Flask框架对外提供接口。该层主要负责向量数据库检索、相关文档重排序、提示词组装、模型推理及流式事件输出。模型服务对外暴露同步与流式两类接口，业务服务可以根据实际场景灵活选择调用方式。将推理服务独立部署后，可在不影响业务API的情况下单独升级模型和检索策略。

数据与索引层中使用关系型数据库保存用户、会话、消息与文档元数据；使用FAISS向量索引存储医学指南等医学资料，并将文本编码为向量构建索引结构以支持向量检索和BM25检索。

\section{系统流程设计}

针对“三高”慢病管理场景，本文将问答流程划分为“输入解析—知识检索—上下文增强—答案生成”四个阶段。系统流程如\ref{fig:system-flowchart}所示。

\vspace{10pt}
\begin{figure}[htb]
    \includegraphics[width=0.8 \textwidth]{image/4/framework.png}
    \caption{系统流程图}
    \label{fig:system-flowchart}
\end{figure}

在输入解析阶段，系统会先接收用户发来的自然语言问句，再结合之前几轮对话的历史信息对查询做语义补全和规范化处理，从而让后续检索能更精准地锁定用户真正的意图。

在知识检索阶段，系统采用混合检索策略，一路是基于向量的语义检索，它依靠Embedding模型把文本映射到高维的向量空间中来表示语义、捕捉深层的语义相似性；另一路则是基于关键词的BM25检索，专门负责精确锁定关键字上的匹配，特别针对于医学领域海量的专业术语。如果只依赖向量检索，有可能会漏掉或误判某些关键医学名词，于是把BM25作为补充手段能有效提高整个检索结果的准确度和覆盖面，两路结果检索完成之后，系统再经过融合、排序和重排这几道处理，筛选出最相关的一批文本片段。

在上下文增强阶段，系统将上一阶段筛选出的文本片段，用户问题和对话历史，通过事先设计好的prompt模板结合，构建成一份结构清晰的输入，让模型在阅读时能轻易分辨出哪一部分来自使用者的提问、哪一部分是从知识库里调出来的参考资料。

在答案生成阶段，模型依据这份组合好的输入来产出回答，以流式方式逐字输出。在生成全部完成后，系统会把检索到的文献信息返还给用户当成参考，同时也会记录下这次输入和输出所消耗的token数量，后者将会跟对话历史一同保存进数据库，为下一次用户提问时构建多轮对话上下文时估算长度提供依据。

\section{业务层实现}

业务层后端服务基于Node.js的框架Express开发，负责用户身份认证与权限控制、会话管理、消息持久化、模型服务编排与知识库运维接口。

\subsection{后端服务架构设计}

系统后端入口位于统一应用服务中，主要路由模块包括用户认证、会话管理、消息管理、模型调用和知识库管理。各模块职责如下：
\begin{enumerate}[label =(\arabic*)]

    \item 认证模块：提供注册与登录接口，完成密码加密与令牌签发；
    \item 会话模块：管理会话创建、查询、更新与软删除；
    \item 消息模块：管理消息读写与 token 信息回填；
    \item 模型模块：接收问题请求，调用本地模型服务并返回生成结果；
    \item 知识库模块：提供文档元信息管理、索引状态查询与重建能力。

\end{enumerate}

\subsection{用户身份认证}

用户身份认证在本系统中不是单独的“登录功能”，而是后续会话管理、权限控制和后台维护操作的统一入口。由于系统同时面向普通用户问答和管理员知识库维护两类场景，后端既要判断用户是否已登录，也要识别用户权限等级，确保不同角色只能访问对应功能。

认证模块主要包括注册、登录和令牌校验三部分。用户注册时，后端接收用户名与密码等基础信息，并在密码加密后写入数据库，避免敏感数据明文存储；用户登录成功后，系统会签发包含身份标识和权限字段的访问令牌，前端在访问受保护接口时需要携带该令牌完成身份校验，后端再根据令牌信息判断请求是否合法以及用户权限范围。通过这一流程，普通用户可正常访问问答能力，知识库管理与索引维护等高权限操作仅向管理员开放，从而在保证使用便利性的同时控制系统安全边界。

系统采用JWT令牌校验机制。前端在请求头中携带Authorization: Bearer <token>后，后端认证中间件会先解析令牌格式，再使用服务端密钥校验签名与有效期，并检查载荷中的用户标识和权限字段。若令牌缺失、签名不合法或已过期，接口会直接返回未授权状态（如401）；若用户身份有效但权限不足，则返回禁止访问状态（如403）。校验通过后，系统将用户信息写入请求上下文，供后续会话、消息和知识库接口复用，以避免重复解析并保证权限判断一致性。

\subsection{会话管理}

会话管理是后端业务逻辑中的核心部分。系统为每个用户维护独立会话空间，并将每轮对话中的角色、正文、时间戳、引用证据和统计信息写入数据库，以支撑历史记录查询和多轮上下文组织。

在具体流程上，当用户提交问题时，后端会先判断当前会话是否存在；若不存在，则创建新会话并写入用户消息；若已存在，则直接在原会话下追加消息。模型返回回答后，系统再将助手回答与证据一并写回，并同步更新会话摘要和更新时间。这样既能保证多轮对话连续性，也方便前端展示最近会话状态。

\subsection{消息流式转发}

为了缩短用户等待时的空白感，后端加入了面向前端的流式转发机制，Node.js服务接收到模型服务返回的增量文本后，会继续通过SSE把内容持续发送到前端，这样回答还没有完全生成时，界面上就已经能看到文本逐步显示出来，整体交互会更连贯。

具体处理时，后端会先把用户消息和助手占位消息写入数据库，再和模型服务建立流式连接，后续只要模型服务持续返回新的文本片段，后端就会把这些内容整理成前端可以解析的事件数据并立即转发出去；等回答生成结束后，再统一回写完整结果和相关统计信息，这样一来，界面能够及时更新，数据库里保存的内容也能保持完整一致。

\subsection{知识库管理接口}

知识库管理接口主要面向管理员使用，任务是把文档维护和索引管理这一类后台操作组织起来，接口内容包括文档元信息查询、文本上传入库、索引状态查看、增量更新和全量重建等，这样知识库相关操作就能放在统一入口中处理，管理起来也更方便。

在知识入库过程中，后端接收到文本内容和来源信息后，会继续调用后续处理流程，依次完成文本清洗、内容切分、Embedding编码和索引写入，写入成功以后，再触发模型服务对索引进行热更新；如果中间某个环节出现问题，系统会把错误状态记录下来，并返回较明确的提示信息，方便后面排查和恢复，这样处理之后，知识更新就不再是彼此分散的单独步骤，而是能和在线问答过程更自然地衔接起来。

\section{数据库设计}

后端数据库使用PostgreSQL，PostgreSQL可支持使用“JSONB内容字段”的存储。在保存用户与模型的对话历史时，用户消息和模型消息、系统消息的结构并不完全相同，例如：模型消息在返回时会携带RAG过程中检索到的资料信息，而用户消息则不需要这个字段。面对这类字段需求不同的情况，JSONB 格式比常规的关系型表设计要灵活得多，能很好地应对问答消息里字段不固定、结构可能时常变动的状况。系统将用户、会话、消息等较为固定的实体信息保存在关系型字段里；将消息正文、引用资料信息、Token消耗等可选字段的内容存放在JSONB字段中。通过这一混合存储方式，就能用同一张表适应模型消息和用户消息的不同格式，不必再去为每一种消息单独建表，或在一张表中留下大量的空字段。  

表\ref{tab:backend-core-tables}中列出了系统中关键的数据表及其主要字段，其中，users表用于存储用户信息，并且为每个用户分配独特的id，用户id是对其他表查询的关键信息；conversations表用于存储会话元信息（标题、更新时间、软删除标记等）；messages表用于保存逐轮对话内容与token统计信息；medical\_documents表用于保存知识文档元数据；knowledge\_index\_meta表用于记录索引状态（dirty/running/synced/error）。

除关系型数据库外，系统还使用FAISS对知识文本的Embedding向量进行独立存储与管理，以支撑RAG流程中的高效向量检索。具体而言，知识文档在完成清洗与切分后，会先由Embedding模型编码为稠密向量，再写入FAISS索引文件中；与此同时，每个文本片段对应的文档标识、分段标识及来源信息仍保存在PostgreSQL中。这样设计的原因在于，向量数据更适合交由FAISS执行相似度计算与近邻检索，而文档元数据、业务实体和状态信息则更适合由关系型数据库统一维护。

在索引结构设计上，系统采用基于内积相似度的IndexFlatIP作为底层索引，并在其外层封装IndexIDMap2，使每一个文本片段向量都能够绑定显式的faiss\_id。通过这种设计，索引不仅能够完成初始批量构建，还支持后续按照ID进行增量维护。当知识库中新增文档时，系统在完成文本切分和向量编码后，可以直接通过add\_with\_ids将新增片段追加到现有索引中，而不必每次都重新构建整个向量库；当文档被删除或更新时，也可以基于片段ID对相应向量执行移除或替换操作。相较于仅适合静态构建的索引方式，这种ID感知的索引结构更适合医学知识库持续更新的应用场景。

为了保证索引文件与业务数据始终保持一致，系统通过knowledge\_index\_meta表记录索引状态、最近重建时间和异常信息，并在文档新增、更新或删除后触发增量更新或全量重建流程。通过这种“FAISS负责向量召回、PostgreSQL负责业务管理”的协同方式，系统既能够满足医学问答对检索效率的要求，也能够保证知识来源的完整性、可追溯性以及后续索引维护的灵活性。

\section{模型推理服务实现}

\subsection{模型服务功能设计}

本文将模型推理部分独立部署为Python服务，并和业务后端解耦。这样设计有两点直接好处，一是可以把GPU资源集中用于检索和推理任务，减少和业务逻辑混合部署带来的资源竞争，二是后续即使替换Embedding模型、重排模型或生成模型版本，也不需要同步改动业务层接口。

模型服务主要承担四个关键任务，一是接收业务层转发的问题、历史对话和生成参数，二是完成查询向量化、候选召回和重排，输出可供生成阶段使用的证据集合，三是按照提示词模板把问题、历史消息和检索证据组织成统一输入，再调用生成模型产出回答，四是以同步或流式方式把结果返回业务层，同时附带引用证据和必要状态信息。经过这样的拆分后，模型服务实际上成为系统中连接“知识检索”和“回答生成”的核心执行节点。

\subsection{模型加载与推理组织}

第三章和第四章已经分别对检索流程、Prompt模板和模型输入组织进行了分析，因此本节主要说明这些方法在系统中的实际加载方式与运行组织方式。考虑到医学问答服务属于在线交互场景，系统若在每次请求到达后再动态加载Embedding模型、重排模型和生成模型，将带来较大的权重读取与显存分配开销，不利于响应时延控制。基于这一考虑，本文采用服务启动时一次性完成模型初始化的方式，将相关模型预先加载到推理环境中，并在服务运行期间保持常驻状态，使在线请求能够直接复用已有推理资源，从而提高系统响应的稳定性。

在生成模型组织方面，系统采用“基础模型+领域适配权重”的加载方式，即以通用大语言模型作为底座，在推理阶段叠加医学问答任务训练得到的LoRA/DPO适配权重。这样既保留了基础模型的通用语言理解与生成能力，又能够体现领域微调后在医学术语表达、回答风格控制和风险边界把握方面的优化效果。与直接部署完整微调模型相比，该方式在存储开销、模型切换和后续迭代上更具灵活性，也更符合当前大模型应用中的参数高效部署思路。

在在线推理过程中，系统接收业务层传入的问题和会话上下文后，先调用已加载的检索相关模型完成证据召回与筛选，再按照设计好的输入模板构造生成上下文，最后由生成模型输出回答结果。

总体来看，这种“预加载+统一编排”的组织方式将模型初始化、检索调用和回答生成纳入同一推理框架之中，有助于减少重复加载带来的性能损耗，并保持在线服务过程的连续性。同时，由于不同模型和处理环节之间边界相对清晰，系统在后续替换Embedding模型、调整重排策略或升级生成模型时，也不需要对整体服务结构进行大幅修改，从而为系统的持续优化和扩展提供了较好的工程基础。

\subsection{接口设计与结果返回}

为了适应不同交互场景，模型服务对外提供同步调用和流式调用两类接口。同步接口更适合调试、批量测试和非实时任务，调用方在一次请求中等待完整回答返回；流式接口更适合在线问答场景，模型服务会把回答拆成连续文本片段逐步输出，让前端能够边接收边展示，减少等待完整回答时的停顿感。

在返回内容上，模型服务不只输出最终回答文本，还会同时返回检索证据、引用来源和必要状态字段。对于流式模式，服务通过事件形式持续返回增量内容，并在结束时统一发送完成标记；如果推理过程中出现异常，则返回错误事件并附带错误信息，便于业务层捕获和提示。通过这种接口设计，模型服务向上对接业务后端时可以保持调用方式统一，向下也能兼容不同推理模式。

\subsection{检索链路调度}

在系统实现上，RAG服务通过统一调度入口管理检索相关步骤。系统不会把BM25检索、向量检索、融合排序和重排分别暴露为分散接口，而是由模型服务在内部完成链路编排，对外只表现为一次“检索增强问答”调用。这样设计可以减少业务层对底层检索细节的依赖，后续即使调整召回方式或更换重排模型，上层接口也不需要同步修改。

在调度过程中，系统先根据用户问题生成检索查询，再并行触发不同召回路径，随后对候选结果做统一编号、去重和排序。完成排序后，系统只保留数量受控的高相关证据进入下游生成流程，避免把大量低价值文本直接送入模型。也就是说，RAG服务在系统中的作用不只是“把资料找出来”，更重要的是在有限上下文窗口内尽可能选出最值得保留的证据。

\subsection{证据封装与结果回传}

完成检索与筛选后，RAG服务还需要把中间结果整理成业务层和前端都能直接使用的结构。基于这一目标，系统会把最终证据统一封装为JSON对象，字段中包含文本内容、来源信息、标题路径、相关性顺序和必要标识。这样处理后，生成模型可以直接使用这些证据构造输入，业务后端也可以在回答完成后把同一份证据回写数据库，前端则能够基于这些字段展示“引用来源”或“检索依据”等信息。

在结果返回阶段，RAG服务输出的不仅是最终回答文本，还会同时返回检索证据、引用元数据和必要状态字段。将检索证据与生成结果作为同一次调用的统一返回内容后，系统能够较好保持“回答内容”和“回答依据”之间的一致性。
